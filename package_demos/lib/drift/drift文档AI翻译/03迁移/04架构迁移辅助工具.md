# 架构迁移辅助工具

借助生成的代码遍历所有数据库架构版本，实现**分步式**数据库迁移编写。

数据库迁移通常采用**增量编写**的方式：通过一段代码将数据库架构升级到下一个版本。通过链式执行这些迁移步骤，即便应用版本非常老旧，你也能完成完整的架构升级。

---

然而，在不同应用版本之间可靠地编写迁移脚本并非易事。这类代码需要长期维护和测试，但数据库架构日益复杂的同时，不应让迁移逻辑也变得繁琐难懂。我们来看一个典型案例，正是这类场景让增量迁移模式变得难以维护：

1. 初始数据库架构中包含多张表；
2. 从版本1升级到版本2时，我们为 `Users` 表添加了 `birthDate` 列；
3. 到了版本3，我们发现完全不需要存储用户数据，于是删除了这张表。

在版本3发布前，版本1→2的迁移可以简单写成 `m.addColumn(users, users.birthDate)`。但如今源代码中已经不存在 `Users` 表了，这段代码**无法再正常运行**！

当然，我们可以手动判断：版本1→2的迁移已经失效，如果用户直接从1升级到3，就跳过该步骤。但这会大幅增加代码复杂度。对于跨多个版本的复杂迁移脚本，这种写法很快会变得难以理解和维护。

---

如果你使用 `make-migrations` 命令或其他 Drift 工具导出了数据库架构，可以让 Drift 为**所有架构版本**生成精简的辅助代码。该功能是 `make-migrations` 命令的一部分，也可以手动单独调用。

生成的文件（与你的数据库文件同级目录）会定义一个 `stepByStep` 工具方法，可直接传入 `onUpgrade` 回调中使用：

```dart
// 这是 make-migrations 默认生成的文件。若手动执行 drift_dev schema steps 命令，
// 你可以自定义生成文件的名称和路径
import 'database.steps.dart';

@DriftDatabase()
class Database extends _$Database {
  // 构造函数和其他方法

  @override
  MigrationStrategy get migration {
    return MigrationStrategy(
      onUpgrade: stepByStep(
        from1To2: (m, schema) async {
          await m.addColumn(schema.users, schema.users.birthdate);
        },
        from2To3: (m, schema) async {
          await m.deleteTable('users');
        },
      ),
    );
  }
}
```

这种方式极大简化了迁移代码的编写，原因如下：

1. 每个 `fromXToY` 函数都能访问**目标版本的架构**。因此，即便后续删除了 `users` 表，在1→2的迁移中，`schema.users` 依然可用；
2. 迁移逻辑被拆分为独立函数，更易于维护；
3. 你可以持续测试针对旧架构版本的迁移，确保迁移逻辑正确，避免不同版本升级的用户遇到异常。

## 自定义分步迁移

`drift_dev schema steps` 命令生成的 `stepByStep` 函数会提供一个 `OnUpgrade` 回调。你可以自定义升级逻辑，例如在迁移完成后添加外键校验。

`Migrator.runMigrationSteps` 辅助方法可实现该需求，示例如下：

```dart
onUpgrade: (m, from, to) async {
  // 迁移期间临时关闭外键约束，迁移完成后重新开启
  await customStatement('PRAGMA foreign_keys = OFF');

  await m.runMigrationSteps(
    from: from,
    to: to,
    steps: migrationSteps(
      from1To2: (m, schema) async {
        // 版本1→2 新增 dueDate 字段
        await m.addColumn(schema.todos, schema.todos.dueDate);
      },
      from2To3: (m, schema) async {
        // 版本2→3 新增 priority 字段
        await m.addColumn(schema.todos, schema.todos.priority);
      },
    ),
  );

  if (kDebugMode) {
    // 调试模式下校验外键是否异常
    final wrongForeignKeys = await customSelect(
      'PRAGMA foreign_key_check',
    ).get();
    assert(
      wrongForeignKeys.isEmpty,
      '${wrongForeignKeys.map((e) => e.data)}',
    );
  }

  await customStatement('PRAGMA foreign_keys = ON;');
},
```

该示例中，迁移前关闭外键约束，迁移完成后重新开启；调试模式下会自动校验外键一致性，快速发现迁移问题。

## 迁移至分步迁移模式

如果你在 Drift 支持 `stepByStep` 之前就已使用该库，或从未导出过数据库架构，可以通过在 `Migrator.runMigrationSteps` 中**固定起始版本号**，切换到分步迁移模式。

你可以先手动完成旧版本的迁移，将数据库升级到分步迁移的**起始版本**，之后所有新版本变更都使用分步迁移实现：

```dart
onUpgrade: (m, from, to) async {
  await customStatement('PRAGMA foreign_keys = OFF');

  // 手动执行旧版本迁移，升级到架构版本2（分步迁移的起始点）
  if (from < 2) {
    // 版本1→2 新增 dueDate 字段（切换分步迁移前的逻辑）
    await m.addColumn(todos, todos.dueDate);
  }

  // 从版本2开始，使用分步迁移
  await m.runMigrationSteps(
    from: math.max(2, from),
    to: to,
    steps: migrationSteps(
      from2To3: (m, schema) async {
        await m.addColumn(schema.todos, schema.todos.priority);
      },
    ),
  );

  if (kDebugMode) {
    final wrongForeignKeys = await customSelect('PRAGMA foreign_key_check').get();
    assert(wrongForeignKeys.isEmpty, '${wrongForeignKeys.map((e) => e.data)}');
  }

  await customStatement('PRAGMA foreign_keys = ON;');
},
```

我们为版本号设置了**下限2**，确保所有旧版本迁移已执行完成。此后，所有架构变更都可以通过生成分步迁移代码实现。

如果不做这一步，直接从版本1升级到版本3的用户，会无法完整执行所有必要的迁移变更。

## 手动生成

### 重要提示

本命令**专门用于生成分步迁移辅助代码**。如果你正在使用 `make-migrations` 命令，该步骤会自动完成，无需手动调用。

Drift 提供了导出旧架构版本的工具。导出所有架构版本后，执行以下命令生成分步迁移辅助代码：

```bash
dart run drift_dev schema steps drift_schemas/ lib/database/schema_versions.dart
```

- 第一个参数 `drift_schemas/`：存储导出架构的文件夹；
- 第二个参数：生成文件的路径（通常与数据库类同级）。

生成的文件包含 `stepByStep` 方法，可直接用于编写迁移逻辑：

```dart
// 本文件由 drift_dev schema steps 命令自动生成
import 'schema_versions.dart';

MigrationStrategy get migration {
  return MigrationStrategy(
    onCreate: (Migrator m) async {
      await m.createAll();
    },
    onUpgrade: stepByStep(
      from1To2: (m, schema) async {
        // 版本1→2 新增 dueDate 字段
        await m.addColumn(schema.todos, schema.todos.dueDate);
      },
      from2To3: (m, schema) async {
        // 版本2→3 新增 priority 字段
        await m.addColumn(schema.todos, schema.todos.priority);
      },
    ),
  );
}
```

`stepByStep` 为每个版本升级提供独立回调，负责执行对应版本的迁移。回调接收两个参数：

1. `m`：迁移器（与原生 `onUpgrade` 回调的迁移器一致）；
2. `schema`：**目标升级版本**的架构对象。
   例如：`from1To2` 中的 `schema` 对应版本2的数据库架构，迁移器会默认适配该版本，调用 `m.recreateAllViews()` 会重建版本2状态下的视图。
