# 导出模式

存储应用的所有数据库模式版本，用于校验。

---

### 重要提示

该命令专门用于导出数据库模式。若你已使用 `make-migrations` 命令，模式导出的操作会**自动完成**，无需手动执行。

按照设计，Drift 的代码生成器仅能识别数据库模式的**当前状态**。修改模式时，将旧版本的模式快照存储到文件中会很有帮助。后续，Drift 工具可通过读取所有模式文件，校验你编写的数据库迁移逻辑。

我们建议**首次导出初始模式**。之后，每一次变更的模式版本（即每次修改数据库中的 `schemaVersion`）都需要进行存储。本文档默认在项目**根目录**创建 `drift_schemas/` 文件夹存放模式文件，目录结构示例如下：

```
my_app
  .../
  lib/
    database/
      database.dart
      database.g.dart
  test/
    generated_migrations/
      schema.dart
      schema_v1.dart
      schema_v2.dart
  drift_schemas/
    drift_schema_v1.json
    drift_schema_v2.json
  pubspec.yaml
```

当然，你也可以根据自身工作流，使用其他文件夹或子目录。

仅靠 `build_runner` 无法完成模式导出及对应代码生成，因此需要按照本文档的说明进行配置。

但我们相信这一切都是值得的！校验迁移逻辑能确保你在修改数据库后不会出现异常。如果操作过程中遇到阻碍，欢迎随时发起相关讨论。

---

## 导出模式

首先，创建第一个模式文件：

```bash
$ mkdir drift_schemas
$ dart run drift_dev schema dump lib/database/database.dart drift_schemas/
```

该命令会让生成器读取 `lib/database/database.dart` 中定义的数据库，并将其模式提取到新建的文件夹中。

修改数据库模式后，**再次运行该命令**即可。例如，我们修改了数据表并将 `schemaVersion` 升级为 2，只需重新执行命令导出新模式：

```bash
$ dart run drift_dev schema dump lib/database/database.dart drift_schemas/
```

每次修改数据库模式并升级 `schemaVersion` 后，都需要执行该命令。

Drift 会自动将文件夹内的文件命名为 `drift_schema_vX.json`，其中 `X` 为数据库当前的模式版本。若 Drift 无法从 `schemaVersion` getter 中提取版本号，可**显式指定完整路径**：

```bash
$ dart run drift_dev schema dump lib/database/database.dart drift_schemas/drift_schema_v3.json
```

## 导出数据库文件

如果你不想导出**数据库类**的模式，而是想导出**已存在的 sqlite3 数据库文件**的模式，也可以实现！`drift_dev schema dump` 命令会将 sqlite3 数据库文件作为第一个参数，并从中提取对应的模式。

导出模式的目标是**匹配构成数据库的实际 CREATE 语句**。因此，布尔值、日期时间值等部分列会以原生 SQL 类型生成。以真实的数据源作为快照，而非使用 Drift 的高层级类型，能让导出的模式更稳定，不受应用或 Drift 变更（可能影响模式生成方式）的干扰。

---

## 下一步操作

将各版本模式导出为文件后，Drift 工具便可生成**兼容多模式版本**的代码。

这一功能支持**分步迁移**：Drift 会为你需要编写的每一次数据库迁移生成样板代码，你只需填写实际变更的部分即可。这会大幅简化迁移代码的编写。

通过识别所有模式版本，Drift 还能生成测试代码，让你轻松为所有数据库迁移编写单元测试。

## 导出模式的问题调试

### 警告

本节详细描述了一种**特定的模式导出问题**，绝大多数用户不会遇到。当 Drift 遇到本节所述问题时，会自动引导你查看该部分。

首先，Drift 是一款代码生成器：它会根据你的定义生成 Dart 类，负责在运行时构建 `CREATE TABLE` 语句。尽管 Drift 仅通过分析代码就能精准识别数据库模式（这也是它生成类型安全代码的核心能力），但部分细节需要**运行 Dart 代码**才能获取。

一个常见的例子是**默认值**：以下定义方式易于分析：

```dart
class Entries extends Table {
  TextColumn get content => text().withDefault(const Constant('test'))();
}
```

但你也可以这样编写代码：

```dart
String computeDefaultContent() {
  // ...
}

class Entries extends Table {
  TextColumn get content => text().withDefault(Constant(computeDefaultContent()))();
}
```

由于 Drift 模式需要**完整表示数据库结构**（包含默认值），因此我们必须运行 `computeDefaultContent` 这类代码片段！

在旧版 Drift 中，该问题的解决方案是直接将源码（如 `computeDefaultContent()`）嵌入模式文件。生成迁移代码时，该代码会调用 `computeDefaultContent`，在运行时还原模式（让 Drift 可将其与实际模式对比，用于测试）。但该方案存在**两个致命缺陷**：

1. 若后续因新版模式不再需要，删除了 `computeDefaultContent` 方法，旧版模式的生成代码会调用一个不存在的方法。
2. 若后续修改了 `computeDefaultContent` 的实现，会隐式变更模式（并需要执行迁移）。但 Drift 模式仅知道需要执行 `computeDefaultContent()`，无法得知该方法在旧版模式中的执行结果。

为解决该问题，Drift 会尝试运行**部分数据库代码**。因此，若在命令行执行 `make-migrations` 或模式导出时，`computeDefaultContent()` 的返回值为 `hello world`，Drift 会在内部重写数据表：

```dart
class Entries extends Table {
  TextColumn get content => text().withDefault(const Constant('hello world'))();
}
```

后续你修改或删除 `computeDefaultContent()`，都不会影响模式测试的正确性。

因此，从命令行运行部分数据库代码是实现高质量模式导出的**必要条件**，但遗憾的是该操作并非毫无问题。你通过 Dart 运行 `drift_dev`，而应用可能导入了 Flutter 专用的API：

```dart
import 'package:flutter/material.dart' show Colors;

class Users extends Table {
  IntColumn get profileBackgroundColor =>
      integer().withDefault(Constant(Colors.red.shade600.value))();
}
```

要计算此处的默认值，需要执行 `Colors.red.shade600.value`。而 `Color` 定义在 `dart:ui` 中（Dart 命令行应用无法使用该库），因此 Drift 无法分析该模式。由于 Dart 的导入机制，在**导入了 Flutter 的 Dart 文件**中定义常量也会引发该问题：

```dart
// constants.dart
import 'package:flutter/flutter.dart';
const defaultUserName = 'name';

// 数据库文件
import 'constants.dart';
class Users extends Table {
  // 存在问题：生成迁移代码时需要导入 constants.dart，而该文件依赖 Flutter
  TextColumn get name => text().withDefault(const Constant(defaultUserName))();
}
```

反之，**仅在数据库文件中导入 Flutter**并不会产生问题：

```dart
import 'package:drift_flutter/drift_flutter.dart';

class Users extends Table {
  // 无问题：withDefault 仅引用 Drift 核心 API（Constant）
  TextColumn get name => text().withDefault(const Constant('name'))();
}
```

Drift 会自动移除**不影响模式**的内容（如 `clientDefault` 或类型转换器），这些内容可以自由导入 Flutter 专用API。

## 问题修复

当 Drift 无法分析你的模式，并抛出异常指向本文档该章节时，通常可以通过**重构代码**解决问题。

首先需要明确问题根源。你可以让 Drift 导出用于运行数据库的内部代码：

```bash
dart run drift_dev make-migrations --export-schema-startup-code=schema_description.dart
```

随后，自行运行该代码复现错误：

```bash
dart run schema_description.dart
```

Dart 编译器会打印出导致引入 Flutter 等不支持库的依赖路径。

明确问题后，即可逐步解决：

1. 检查 Drift 导入 Flutter 专用文件的原因：是否是默认值中使用了该文件定义的方法或字段？
2. 尝试重构代码，将这些定义移至**未导入 Flutter**的文件中。
3. 重复操作直至问题解决！

我们希望该问题不会影响绝大多数用户，但它确实是一个较难解决的问题。如需更多指导，请在对应问题下留言评论。

---

### 总结

1. 核心用途：导出 Drift 数据库的多版本模式，用于校验迁移、生成测试代码；
2. 核心命令：`dart run drift_dev schema dump`，每次升级 `schemaVersion` 后必须执行；
3. 常见问题：数据库代码依赖 Flutter API 会导致导出失败，通过重构代码分离 Flutter 依赖即可修复。
