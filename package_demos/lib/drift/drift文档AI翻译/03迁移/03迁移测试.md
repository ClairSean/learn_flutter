# 迁移测试

生成测试代码，为你的数据库迁移编写单元测试

---

### 重要提示

如果你正在使用 `make-migrations` 命令，**测试代码会自动生成**，无需手动编写。

虽然可以手动编写迁移逻辑，但专用的迁移测试工具能确保迁移逻辑正确、**不会丢失任何数据**。

Drift 的迁移测试工具遵循以下流程：

1. 每次修改数据库结构后，使用工具将当前结构导出为独立文件；
2. 使用 Drift 工具生成测试代码，验证迁移能将数据库升级为预期结构；
3. 使用生成的代码简化迁移编写工作。

本文介绍步骤 2 和 3，默认你已完成步骤 1（修改结构时导出结构文件）。你也可以参考 Drift 官方的单元测试通用文档。

---

## 编写测试

将数据库结构文件导出到指定文件夹后，可基于这些文件生成**旧版本的数据库类**。
为了验证迁移，Drift 会生成一个精简的数据库实现，**仅用于迁移测试**。

你可以将测试代码放在任意目录，推荐放在 `test/` 的子目录中。
例如，将代码生成到 `test/generated_migrations/`，执行以下命令：

```bash
dart run drift_dev schema generate drift_schemas/ test/generated_migrations/
```

配置完成后，就可以编写测试了！示例如下：

```dart
import 'package:test/test.dart';
import 'package:drift_dev/api/migrations_native.dart';

// 导入之前生成的测试目录
import 'generated_migrations/schema.dart';

void main() {
  late SchemaVerifier verifier;

  setUpAll(() {
    // GeneratedHelper() 由 Drift 自动生成，verifier 是 drift_dev 提供的验证API
    verifier = SchemaVerifier(GeneratedHelper());
  });

  test('从 v1 版本升级到 v2 版本', () async {
    // startAt(1)：创建包含 v1 版本所有表的数据库连接
    final connection = await verifier.startAt(1);
    final db = MyDatabase(connection);

    // 执行迁移到 v2，并验证数据库结构符合预期
    await verifier.migrateAndValidate(db, 2);
  });
}
```

### 通用测试流程

1. **初始化旧结构**：使用 `verifier.startAt()` 创建指定旧版本结构的数据库连接，包含该版本的所有表、索引和触发器；
2. **创建测试数据库**：为你的数据库类添加一个接收 `QueryExecutor` 的构造函数，将验证器的连接传入，创建测试用数据库实例；
3. **执行迁移验证**：调用 `verifier.migrateAndValidate(db, 目标版本)`，该方法会执行你编写的迁移逻辑，并校验最终结构。

`migrateAndValidate` 会从 `sqlite_schema` 表中提取所有 `CREATE` 语句，**语义化对比结构**。若发现结构不匹配，会抛出 `SchemaMismatch` 异常，导致测试失败。

---

## 编写可测试的迁移

若需要测试**向旧版本升级**的迁移（例如当前版本是 v3，测试 v1→v2），你的 `onUpgrade` 处理程序必须支持升级到低于当前 `schemaVersion` 的版本。

解决方案：

- 在 `onUpgrade` 回调中判断 `to` 参数，按需执行不同迁移逻辑；
- 直接使用**分步迁移（step-by-step migrations）**，该功能会自动处理版本适配。

---

## 验证数据完整性

除了校验表结构变化，还需要确保**迁移前的数据在迁移后依然存在**。
你可以使用 `schemaAt` 获取 `sqlite3` 原生数据库连接，**在迁移前插入测试数据**，迁移后校验数据是否完整。

> 注意：不能使用应用的正式数据库类测试，因为它始终基于最新结构解析数据。
> 你可以让 Drift 生成**旧版本的数据类和伴随类**，用于此场景。

开启该功能，需在生成命令中添加 `--data-classes` 和 `--companions` 参数：

```bash
dart run drift_dev schema generate --data-classes --companions drift_schemas/ test/generated_migrations/
```

生成后，通过别名导入旧版本类：

```dart
import 'generated_migrations/schema_v1.dart' as v1;
import 'generated_migrations/schema_v2.dart' as v2;
```

使用旧版本类创建、校验数据的完整示例：

```dart
void main() {
  // ...
  test('从 v1 升级到 v2 后数据完整', () async {
    final schema = await verifier.schemaAt(1);

    // 向旧版本数据库插入测试数据
    final oldDb = v1.DatabaseAtV1(schema.newConnection());
    await oldDb.into(oldDb.todos).insert(
          v1.TodosCompanion.insert(
            title: '第一条待办',
            content: '迁移后应该仍存在',
          ),
        );
    await oldDb.close();

    // 执行迁移并验证结构
    final db = MyDatabase(schema.newConnection());
    await verifier.migrateAndValidate(db, 2);
    await db.close();

    // 校验迁移后数据是否存在
    final migratedDb = v2.DatabaseAtV2(schema.newConnection());
    final entry = await migratedDb.select(migratedDb.todos).getSingle();
    expect(entry.id, 1);
    expect(entry.dueDate, isNull); // 迁移设置的默认值
    await migratedDb.close();
  });
}
```

## 运行时验证数据库结构

除了编写单元测试，`drift_dev` 还提供了**原生平台运行时结构验证API**，无需额外配置即可校验当前数据库结构。

使用方式：

```dart
// 导入迁移测试工具
import 'package:drift_dev/api/migrations_native.dart';

class MyDatabase extends _$MyDatabase {
  @override
  MigrationStrategy get migration => MigrationStrategy(
        onCreate: (m) async {
          /* 初始化逻辑 */
        },
        onUpgrade: (m, from, to) async {
          /* 你的迁移逻辑 */
        },
        beforeOpen: (details) async {
          // 原有逻辑：启用外键等

          // 仅在调试模式下验证结构（避免打包冗余代码）
          if (kDebugMode) {
            await validateDatabaseSchema();
          }
        },
      );
}
```

---

### 核心总结

1. **自动生成**：使用 `make-migrations` 可一键生成迁移测试代码；
2. **结构验证**：`SchemaVerifier` 可验证旧版本→新版本的迁移结构正确性；
3. **数据校验**：生成旧版本数据类，验证迁移后数据不丢失；
4. **运行时检查**：调试模式下可实时校验数据库结构，提前发现问题。
