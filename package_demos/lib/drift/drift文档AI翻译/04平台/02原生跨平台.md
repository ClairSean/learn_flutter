# 原生 Drift（跨平台）

在移动设备与桌面端运行 Drift

---

## 支持的平台

`drift/native.dart` 库依托 `sqlite3` 包发送查询。自 Drift 2.32.0 版本起，**所有原生 Dart 和 Flutter 平台均无需额外配置或依赖**即可直接使用。

旧版本需要依赖 `sqlite3_flutter_libs`（或 Flutter 之外的自定义方案），**现已不再需要**。

---

## Drift 托管的后台隔离区

作为 C 语言库，SQLite 会**同步执行** SQL 语句，执行 IO 操作或计算逻辑时会阻塞发起请求的线程。尤其在移动应用中，这种阻塞特性意味着**不能在 UI 隔离区直接访问数据库**，否则会导致掉帧或其他 UI 异常。

使用 `NativeDatabase.createInBackground` 替代原生 `NativeDatabase` 构造函数时，Drift 会创建一个**后台隔离区**专门托管数据库：

```dart
QueryExecutor openDatabase() {
  return NativeDatabase.createInBackground(File('path/to/database.db'));
}
```

你可以将返回的 `QueryExecutor`（查询执行器）传入数据库类的构造函数中。这意味着**数据库的使用方式完全不变**，仅需修改初始化代码，适配后台隔离区即可。

---

## 使用多个读隔离区

对于绝大多数应用，使用**单个后台隔离区**托管数据库已足够。但在部分场景下，为数据库配置多个后台隔离区会更有优势：

1. 若应用启动时需要执行大量查询，多隔离区可提升启动性能；
2. 若同时存在「耗时查询」（如大表数据读取）和「快速轻量查询」，将查询分发到多个隔离区，可避免长耗时查询影响其他操作；
3. 单线程模式下，长耗时写入或事务会阻塞读取操作；多隔离区则无此问题。

高效使用多隔离区需要开启 **WAL（预写日志）** 模式，该模式允许**一个写入器 + 多个读取器**并行操作同一个数据库文件。**未开启 WAL 时，多隔离区访问同一数据库会触发「数据库已锁定」错误**。

在 `NativeDatabase.createInBackground` 中通过 `readPool` 参数可启用额外的读隔离池：

```dart
QueryExecutor openMultiThreadedDatabase() {
  return NativeDatabase.createInBackground(
    File('path/to/database.db'),
    setup: (database) {
      // 关键配置：多线程访问数据库必须开启 WAL，否则会触发「数据库已锁定」错误
      // 开启 WAL 后，单写入器 + 多读取器可并行操作数据库
      database.execute('pragma journal_mode = WAL;');
    },
    readPool: 4,
  );
}
```

上述代码中，Drift 会创建 **5 个隔离区** 托管数据库：1 个专用写入隔离区 + 4 个只读隔离区。注意：数据库的事务和独占操作始终会使用写入隔离区。

## 通过原生 Drift 使用现有数据库

若你的现有 SQLite 数据库已存储为文件，直接使用 `NativeDatabase(thatFile)` 即可，**无需额外修改**。

若需要从**应用资源包**或其他来源加载数据库，可使用 `LazyDatabase`：它允许你在打开数据库前执行异步操作：

```dart
// 原写法
NativeDatabase(File('...'));

// 新写法
LazyDatabase(() async {
  final file = File('...');
  if (!await file.exists()) {
    // 从资源包、网络或其他来源复制数据库文件
  }
  return NativeDatabase(file);
});
```

关于现有数据库的使用，可参考官方示例获取更详细的说明。

## Android 平台使用的编译选项

在原生平台，默认配置下的 `sqlite3` 包会内置定制版 SQLite，而非使用系统自带的 SQLite。所选编译选项会移除 Drift 未使用的功能，以减小包体积。**重要选项已加粗标注**：

- 启用 `-O3` 性能优化参数
- **`SQLITE_DQS=0`**：禁止 SQLite 接受双引号字符串（将其解析为标识符），与 Drift 和编译后查询的行为保持一致
- **`SQLITE_THREADSAFE=0`**：绝大多数 Flutter 应用仅使用单个隔离区，因此关闭线程安全；仍可通过隔离区 API 执行后台操作，只要所有数据库访问都来自同一线程，就不会出现问题
- `SQLITE_DEFAULT_MEMSTATUS=0`：Drift 未暴露 `sqlite3_status()` 接口，无需启用该功能
- `SQLITE_MAX_EXPR_DEPTH=0`：关闭 SQLite 表达式解析的最大深度限制，提升解析器速度
- `SQLITE_OMIT_AUTHORIZATION`、`SQLITE_OMIT_DECLTYPE`、`SQLITE_OMIT_DEPRECATED` 等：禁用 Drift 不支持的功能
- `SQLITE_USE_ALLOCA`：在栈上分配临时内存
- `SQLITE_UNTESTABLE`：移除仅用于 SQLite 测试的工具函数
- `SQLITE_HAVE_ISNAN`：使用系统自带的 `isnan` 函数，而非 SQLite 内置版本
- **`SQLITE_ENABLE_FTS5`**：启用 FTS5 全文检索引擎
- **`SQLITE_ENABLE_JSON1`**：启用 JSON1 扩展，支持 SQL 查询中的 JSON 操作

SQLite 编译选项的更多细节，可查阅官方文档。

## 仅 Drift 支持的函数

`NativeDatabase` 内置了**标准 SQLite 不提供**的扩展 SQL 函数：

1. **`pow(底数, 指数)` / `power(底数, 指数)`**：接收两个数值参数，返回底数的指数幂；参数非数值或为 `null` 时返回 `null`，行为与 `dart:math` 中的 `pow` 完全一致
2. **`sqrt`/`sin`/`cos`/`tan`/`asin`/`acos`/`atan`**：接收单个参数；参数为 `null` 或非数值时返回 `null`，否则返回 `dart:math` 对应函数的计算结果
3. **`regexp`**：封装 Dart 的 `RegExp` API，`foo REGEXP bar` 等价于 `RegExp(bar).hasMatch(foo)`；注意：每次 SQL 正则调用都会新建 `RegExp` 实例，大查询下可能影响性能
4. **`current_time_millis`**：返回当前 Unix 时间戳（毫秒），等价于 Dart 中的 `DateTime.now().millisecondsSinceEpoch`

注意：`NaN`、负无穷、正无穷在 SQL 中会被表示为 `NULL`。

在构建选项中启用 `moor_ffi` 模块后，代码生成器会允许你在 Drift 文件或编译查询中使用这些函数。

若要在 Dart 中调用这些方法，需要导入 `package:drift/extensions/native.dart`，使用示例：

```dart
import 'package:drift/drift.dart';
// 该导入为专属扩展：仅搭配 NativeDatabase 可用
import 'package:drift/extensions/native.dart';

class Coordinates extends Table {
  RealColumn get x => real()();
  RealColumn get y => real()();
}

// 使用扩展函数查询
Future<List<Coordinate>> findNearby(Coordinate center, int radius) {
  return (select(coordinates)..where((other) {
    // 计算距离：sqrt((中心点x - 当前x)² + (中心点y - 当前y)²) < 半径
    final distanceSquared = sqlPow(center.x - row.x, 2) + sqlPow(center.y - row.y, 2);
    return sqlSqrt(distanceSquared).isLessThanValue(radius);
  })).get();
}
```

其他函数命名规则一致（`sqlSin`、`sqlCos`、`sqlAtan` 等），前缀 `sql` 用于避免与 `dart:math` 中的函数冲突。

## 从 moor_flutter 迁移至 drift/native

### 第一步：修改 `pubspec.yaml`

移除 `moor_flutter` 依赖，添加 `drift`：

```yaml
dependencies:
  drift: ^2.32.1
  sqflite: ^1.1.7 # 仍用于获取数据库存储路径
dev_dependencies:
  drift_dev: ^2.32.1
```

### 第二步：修改导入语句

1. 创建 `FlutterQueryExecutor` 的文件中：将 `moor_flutter` 导入替换为 `package:drift/native.dart`
2. 其他导入过 `moor_flutter` 的文件：统一改为 `package:drift/drift.dart`

### 第三步：替换查询执行器

原代码：

```dart
FlutterQueryExecutor.inDatabaseFolder(path: 'db.sqlite')
```

替换为：

```dart
import 'package:sqflite/sqflite.dart' show getDatabasesPath;
import 'package:path/path.dart' as p;

LazyDatabase(() async {
  final dbFolder = await getDatabasesPath();
  final file = File(p.join(dbFolder, 'db.sqlite'));
  return NativeDatabase(file);
})
```

### 注意事项

- 若你**从未发布过基于 `moor_flutter` 的版本**，可移除 `sqflite` 依赖，改用跨桌面端支持的 `path_provider`；
- 注意：Android 平台上，`FlutterQueryExecutor.inDatabaseFolder` 与 `path_provider` 的存储路径可能不同，**已发布过 `moor_flutter` 版本的应用，必须使用 `sqflite` 的 `getDatabasePath`**，避免数据丢失。

---

### 总结

1. **核心优势**：Drift 2.32.0+ 原生支持全平台，无需额外依赖，后台隔离区完美解决 UI 阻塞问题；
2. **多隔离区**：配合 WAL 模式 + `readPool`，实现高性能并行读写；
3. **兼容适配**：支持现有数据库、资源包加载，提供专属 SQL 扩展函数；
4. **迁移指南**：一键替换依赖、导入和执行器，即可从 moor_flutter 平滑升级。
