# 自适应设计最佳实践

以下是自适应设计的部分推荐最佳实践总结。

## 设计思路

### 拆分你的组件

在设计应用时，尽量将庞大、复杂的组件拆分成更小、更简单的组件。

通过重构组件、复用核心代码片段，可以降低实现自适应 UI 的复杂度。除此之外还有其他好处：

- 在性能方面，大量小型的 const 组件，相比庞大复杂的组件，能提升重建速度。
- Flutter 可以复用 const 组件实例，而大型复杂组件每次重建都需要重新构建。
- 从代码健康度来看，将 UI 拆分成更小的单元，有助于降低每个组件的复杂度。复杂度更低的组件可读性更高、更容易重构，也更不容易出现意料之外的行为。

想了解更多，可以查看《通用方案》章节中自适应设计的三个步骤。

### 基于不同设备形态的优势进行设计

除了屏幕尺寸之外，你还应当花时间考虑不同设备形态独有的优势与短板。让你的多平台应用在所有平台上提供完全一致的功能，并不总是最优方案。你可以思考：在某些设备类别上，聚焦特定能力、甚至移除部分功能是否更合理。

例如：移动设备便携且带有摄像头，但并不适合进行精细的创作类工作。基于这一点，移动端 UI 可以更侧重内容拍摄、添加位置标签等功能；而平板或桌面端 UI 则更侧重内容整理与编辑。

另一个例子是利用网页极低的分享门槛。如果你要部署网页应用，确定需要支持哪些深度链接，并在设计导航路由时将其纳入考量。

核心要点是：思考每个平台最擅长做什么，看看能否利用其独有的功能特性。

### 优先满足触控体验

打造优秀的触控 UI，往往比传统桌面端 UI 更困难，部分原因在于触控设备缺少右键、滚轮、键盘快捷键这类高效输入方式。

应对这一挑战的一种思路是：先聚焦打造体验出色的触控优先 UI。你依然可以使用桌面端目标平台进行大部分测试，以获得更快的迭代速度。但要记得频繁切换到移动设备，验证交互手感是否自然。

在把触控界面打磨完善后，再为鼠标用户调整视觉密度，随后补充各类额外输入方式。把这些其他输入方式当作**效率增强手段**——让操作更快捷的替代方案。重要的是考虑用户使用特定输入设备时的预期，并在应用中体现出来。

## 实现细节

### 不要锁定应用屏幕方向

一款自适应应用应当在不同尺寸、不同形状的窗口上都呈现良好效果。尽管在手机上把应用锁定为竖屏可以缩小最小可行产品的适配范围，但会增加后续实现自适应的成本。

例如，认为手机只会以全屏竖屏模式渲染应用，这个假设并不成立。多窗口应用支持越来越普遍，而折叠屏有很多场景适合多应用并排运行。

如果你**实在必须**锁定应用为竖屏（不推荐这么做），请使用 Display API，而非 MediaQuery 这类方式来获取屏幕物理尺寸。

总结：

- 锁定屏幕方向对部分用户而言会构成无障碍使用问题
- Android 大屏设备支持等级中，最低要求也需要同时支持竖屏与横屏
- Android 设备可以强制覆盖屏幕锁定设置
- 苹果设计规范建议尽量同时支持横竖屏

### 避免基于设备方向进行布局

避免在组件树顶层使用 MediaQuery 的 orientation 字段或 OrientationBuilder 来切换应用布局。这与不通过设备类型判断屏幕尺寸的指导原则类似。设备方向并不一定能反映应用窗口的可用空间大小。

取而代之的是，使用《通用方案》章节中提到的 `MediaQuery.sizeOf` 或 `LayoutBuilder`，并使用 Material 推荐的自适应断点。

### 不要占满全部横向空间

在大屏设备上运行时，将窗口整个宽度用于展示容器或文本框的应用，体验并不友好。

想了解如何避免该问题，可以查看《使用 GridView 进行布局》章节。

### 避免判断硬件类型

在做布局决策时，避免编写代码判断当前运行设备是“手机”“平板”或其他设备类型。

应用实际可用于渲染的空间，并不总是与设备物理全屏尺寸绑定。Flutter 可以运行在多种平台上，你的应用可能在 ChromeOS 的可调整窗口中运行、在平板多窗口模式下与其他应用并排显示，甚至在手机上以画中画模式运行。因此，设备类型与应用窗口尺寸并没有强关联。

取而代之的是，使用 `MediaQuery` 获取应用当前运行的窗口尺寸。

这不仅对 UI 代码有帮助。想了解抽象设备能力如何优化业务逻辑代码，可以观看 2022 年 Google I/O 演讲《Flutter lessons for federated plugin development》。

### 支持多样化输入设备

应用应当支持基础的鼠标、触控板以及键盘快捷键。最常用的用户流程应支持键盘导航，以保障无障碍使用体验。尤其要让应用在大屏设备上遵循键盘无障碍最佳实践。

Material 库提供的组件，在触控、鼠标、键盘交互上都具备优秀的默认行为。

想了解如何为自定义组件添加这类支持，可以查看《用户输入与无障碍适配》章节。

### 恢复列表状态

若要在设备方向改变、列表布局不变的情况下保持滚动位置，可以使用 `PageStorageKey` 类。`PageStorageKey` 会在组件销毁后将其状态持久化存储，并在组件重建时恢复状态。

你可以在 Wonderous 应用中看到相关示例，该应用在 `SingleChildScrollView` 组件中保存了列表状态。

如果列表组件在设备方向改变时会切换布局，你可能需要通过一些计算来在屏幕旋转时调整滚动位置。

### 保存应用状态

当设备旋转、窗口尺寸变化、折叠或展开时，应用应当保留或恢复应用状态。默认情况下，应用应当维持状态不丢失。

如果你的应用在设备配置变更期间丢失状态，请检查应用使用的插件和原生扩展是否支持大屏等设备类型。部分原生扩展可能在设备形态变化时丢失状态。

想了解该问题的真实案例，可以查看 Medium 上免费文章《为大屏开发 Flutter 应用》中的《问题：折叠/展开导致状态丢失》一节。
