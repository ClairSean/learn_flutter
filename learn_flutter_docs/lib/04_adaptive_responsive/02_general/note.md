# 自适应应用的通用方法

关于如何打造 Flutter 自适应应用的通用建议

那么，该如何将一款为传统移动设备设计的应用，优化得能在各类设备上都呈现精美效果？具体需要哪些步骤？

拥有大型应用适配经验的谷歌工程师，推荐以下**三步适配方案**。

## 第一步：抽象（Abstract）

### 提取共享数据并做抽象处理

首先，确定你需要实现动态适配的组件。分析这些组件的构造函数，提取出可共享的数据。

**常见需要适配的组件**：

- 弹窗类：全屏弹窗、模态弹窗
- 导航类：导航轨、底部导航栏
- 自定义布局类：如判断 UI 区域是高大于宽，还是宽大于高

**示例**：

- 在 `Dialog` 弹窗组件中，可共享弹窗的内容信息。
- 再比如：应用窗口较窄时使用 `NavigationBar`（底部导航栏），窗口较宽时切换为 `NavigationRail`（侧边导航轨）。这两类组件通常共享一组导航目的地数据。这种情况下，你可以创建一个 `Destination` 组件来承载这些数据，为其配置图标和文字标签。

完成数据抽象后，接下来需要判断屏幕尺寸，以确定 UI 的展示方式。

## 第二步：测量（Measure）

### 选择 `MediaQuery.sizeOf` 或 `LayoutBuilder`

有两种方式可获取展示区域的尺寸：`MediaQuery` 和 `LayoutBuilder`。

### MediaQuery

过去，你可能使用 `MediaQuery.of` 获取设备屏幕尺寸。但如今设备屏幕的尺寸与形态多种多样，这种方式可能会产生误导。

举个例子：你的应用在大屏设备上仅以小窗口运行。若使用 `MediaQuery.of` 方法误判为小屏幕，且对应用做了**竖屏锁定**，会导致应用窗口被固定在屏幕中央，四周出现黑边。这在大屏设备上绝非理想的 UI 效果。

**提示**
Material 设计规范建议**不要锁定应用为竖屏**（不要禁用横屏模式）。若你确实需要锁定，至少保证竖屏模式支持正向与反向两种显示方式。

注意：`MediaQuery.sizeOf` 返回的是**应用整个窗口**的当前尺寸，而非单个组件的尺寸。

你可以根据需求选择两种尺寸测量方式：

- 需获取**全局应用窗口**尺寸 → 使用 `MediaQuery.sizeOf`
- 需获取**局部组件**尺寸 → 使用 `LayoutBuilder`

若希望组件全屏展示（即便应用窗口很小），使用 `MediaQuery.sizeOf`，基于应用窗口尺寸选择对应 UI。上文中基于全局窗口做尺寸适配的场景，就适合用 `MediaQuery.sizeOf`。

### 为什么推荐用 `MediaQuery.sizeOf` 替代 `MediaQuery.of`？

此前的建议是使用 `MediaQuery` 的 `of` 方法获取应用窗口尺寸，为何现在方案变更？
**简短答案：性能优化**。

`MediaQuery` 包含大量数据，若你仅需获取尺寸属性，使用 `sizeOf` 方法效率更高。两种方法均以**逻辑像素**（也叫密度无关像素）返回应用窗口尺寸。逻辑像素在不同设备上的视觉尺寸基本一致，是最适配的尺寸单位。
出于同样的性能原因，`MediaQuery` 类也为其他单个属性提供了专用的获取方法。

在 `build` 方法中调用 `MediaQuery.sizeOf(context)` 获取窗口尺寸，会让传入的 `BuildContext` 在尺寸发生变化时自动重建组件。

### LayoutBuilder

`LayoutBuilder` 与 `MediaQuery.sizeOf` 作用相似，但有核心区别：

- `LayoutBuilder` 不返回应用窗口尺寸，而是获取**父组件传递的布局约束**。
- 这意味着它能获取组件树中当前节点的**局部空间尺寸**。
- 同时，`LayoutBuilder` 返回的是 `BoxConstraints`（盒约束）对象，而非固定的 `Size` 对象，会给出内容的有效宽高范围（最小值、最大值），对自定义组件非常实用。

**示例**：
自定义一个组件，希望其尺寸仅依据自身分配到的空间，而非全局应用窗口。这种场景就适合使用 `LayoutBuilder`。

## 第三步：分支（Branch）

此步骤需要确定**尺寸断点**，根据断点选择对应的 UI 版本展示。

**示例**：
Material 布局规范建议：

- 窗口宽度小于 600 逻辑像素 → 使用底部导航栏
- 窗口宽度大于等于 600 逻辑像素 → 使用侧边导航轨

再次强调：UI 选择的依据是**设备可用窗口尺寸**，而非设备类型。
