# 功能与策略

学习如何让你的应用适配平台、应用商店、公司等主体所要求的功能与策略。

大多数实际商用应用都需要适配不同设备与平台的功能和策略。本页面提供在代码中处理这些场景的官方建议。

---

## 基于各类设备的优势进行设计

思考不同设备独有的优势与短板。除了屏幕尺寸、触控/鼠标/键盘等输入方式外，你还能利用哪些独有的功能？Flutter 让你的代码可以在不同设备上运行，但优秀的设计远不止让代码运行起来。思考每个平台的最佳特性，探索可以利用的独有功能。

例如：苹果 App Store 和谷歌 Play 商店对应用有着不同的合规规则。不同的宿主操作系统，在不同时期以及彼此之间，也拥有不同的功能支持。

另一个例子是利用网页极低的分享门槛。如果你正在部署网页应用，确定需要支持哪些深度链接，并在设计导航路由时将这些因素考虑在内。

针对基于这些独有功能实现不同行为的场景，Flutter 推荐的设计模式是：为你的应用创建一组**功能（Capability）**和**策略（Policy）**类。

---

## 功能（Capabilities）

功能用于定义**代码或设备能够做什么**。功能的示例包括：

- 某个 API 是否可用
- 操作系统强制的限制条件
- 物理硬件要求（如摄像头）

## 策略（Policies）

策略用于定义**代码应该做什么**。

策略的示例包括：

- 应用商店指南
- 设计偏好
- 关联宿主设备的资源或文案
- 服务端启用的功能

---

## 如何组织策略代码

最简单的直接方式是使用 `Platform.isAndroid`、`Platform.isIOS` 和 `kIsWeb`。这些 API 可以直接告诉你代码运行在哪个平台，但随着应用支持的平台扩大、宿主平台新增功能，这种方式会出现一些问题。

以下是为应用开发功能与策略的最佳实践：
**避免**使用 `Platform.isAndroid` 等方法来做布局决策，或假设设备的能力。

取而代之的是：在一个方法中描述你要基于什么条件进行分支判断。

### 示例：

你的应用包含一个网页购买链接，但由于策略原因，不希望在 iOS 设备上显示该链接。

```dart
bool shouldAllowPurchaseClick() {
  // 苹果应用商店指南禁止该行为
  return !Platform.isIOS;
}

...
TextSpan(
  text: '在浏览器中购买',
  style: TextStyle(color: Colors.blue),
  recognizer: shouldAllowPurchaseClick
      ? (TapGestureRecognizer()..onTap = () { launch('指定网址'); })
      : null,
)
```

增加这一层封装的好处：代码会清晰说明分支逻辑的原因。这个方法可以直接写在类中，若其他代码也需要相同判断，建议将代码封装到独立类中。

### policy.dart

```dart
class Policy {
  bool shouldAllowPurchaseClick() {
    // 苹果应用商店指南禁止该行为
    return !Platform.isIOS;
  }
}
```

将代码封装到类中后，任何组件测试都可以**模拟** `Policy().shouldAllowPurchaseClick` 的返回值，独立验证行为，不受设备运行环境的影响。这也意味着，后续若调整业务逻辑，只需修改该方法实现，测试代码无需改动。

## 功能（Capabilities）

有时你希望代码实现某个功能，但对应的 API 不存在，或依赖的插件功能尚未在所有支持平台实现。这属于**设备能力上的限制**。

这些场景与策略决策相似，但归类为**功能**。即便类结构相似，也要将策略和功能分开的原因：
Flutter 团队在商用应用实践中发现：区分应用**能做什么**和**应该做什么**，能帮助大型产品在代码编写完成后，轻松应对平台能力变化、规则更新和自身需求调整。

**示例**：
某个平台新增权限要求，调用敏感 API 前需要用户授权。你的团队为平台1完成适配，创建了 `requirePermissionDialogFlow` 功能。后续平台2也新增同类要求时，只需修改该功能的实现，判断 API 版本即可**直接复用已有代码**。

---

## 策略（Policies）

建议你从项目初期就创建 `Policy` 类，即便你认为暂时不需要太多策略判断。随着类的复杂度提升、判断条件增多，可根据功能拆分策略类。

策略的实现支持三种方式：

1. **编译期检查**
   适用于规则几乎不会改变、修改会造成严重后果的场景。例如：平台禁止链接到应用商店、要求使用指定支付服务商。

2. **运行时检查**
   适用于判断设备能力的场景。例如：检测设备是否支持触控屏。

3. **远程过程调用（RPC）**
   适用于功能增量发布、后续可能调整的决策。

---

## 总结

- 使用 **`Capability` 类** 定义代码**能够**做什么。可检查 API 可用性、系统限制、硬件要求（摄像头等），通常包含编译期/运行时检查。
- 使用 **`Policy` 类**（复杂场景可拆分多个类）定义代码**应该**做什么。用于遵守应用商店规则、设计偏好、平台专属资源，支持混合检查方式。
- 通过**模拟功能和策略**测试分支代码，功能/策略变更时，测试代码无需修改。
- 为方法命名时，基于**业务判断逻辑**，而非设备类型。
