# SafeArea 与 MediaQuery

学习如何使用 SafeArea 和 MediaQuery 构建自适应应用。

本页讲解 SafeArea 与 MediaQuery 组件的**使用方式**及**适用场景**。

---

## SafeArea

当你在最新款设备上运行应用时，可能会遇到部分界面被屏幕上的挖孔区域遮挡的情况。
你可以通过 **SafeArea** 组件解决该问题：它会为其子组件添加内边距，避开各类侵入式区域（如刘海、摄像头挖孔）、操作系统界面（例如 Android 状态栏），以及物理屏幕的圆角。

如果你不想要这种默认行为，SafeArea 支持单独禁用上、下、左、右任意一侧的内边距。默认情况下，四个方向均会生效。

通常建议先将 **Scaffold 的 body** 用 SafeArea 包裹，这是一个稳妥的起步做法，但并非必须把它放在组件树这么顶层的位置。

举个例子：如果你有意让应用内容延伸到挖孔区域下方，可以把 SafeArea 下移，只包裹有必要保护的内容，而让应用的其余部分铺满整个屏幕。

使用 SafeArea 可以确保你的应用内容不会被屏幕物理特征或系统界面截断，即便未来市面上出现外形、挖孔样式各异的新设备，你的应用依然能良好适配。

SafeArea 为何能用少量代码实现这么多能力？其底层依赖的是 **MediaQuery** 对象。

---

## MediaQuery

正如 SafeArea 一节所讲，MediaQuery 是构建自适应应用的强大组件。
有些场景你会直接使用 MediaQuery，有些场景则使用 SafeArea——而它底层正是依赖 MediaQuery 实现的。

MediaQuery 能提供大量信息，包括应用当前的窗口尺寸。
它还会暴露各类无障碍设置，例如高对比度模式、文字缩放，或是用户是否正在启用 TalkBack、VoiceOver 这类无障碍服务。
MediaQuery 同样包含设备屏幕相关的特性信息，比如设备是否带有铰链或折叠屏。

SafeArea 依靠来自 MediaQuery 的数据，计算出需要为其子组件设置的内边距大小。
具体来说，它使用的是 MediaQuery 的 **padding** 属性——该属性大致表示屏幕中被系统界面、挖孔或状态栏部分遮挡的区域大小。

那么，为什么不直接使用 MediaQuery 呢？

原因在于：相比直接使用原始的 MediaQueryData，SafeArea 做了一个巧妙的优化，让它更具实用价值。
具体来说，它会修改向其子级暴露的 MediaQuery 数据，让 SafeArea 自身添加的内边距“看起来并不存在”。
这意味着你可以嵌套使用多个 SafeArea，**只有最外层的 SafeArea 会添加避让刘海与系统界面所需的内边距**。

随着应用规模变大、组件频繁调整位置，即便使用多个 SafeArea，你也不用担心内边距被重复叠加；
而如果直接使用 MediaQueryData.padding，就很容易出现边距过大的问题。

你可以用 SafeArea 包裹 Scaffold 的 body，但不必把它放在组件树这么高的层级。
SafeArea 只需要包裹那些被前述硬件特征遮挡后会造成信息丢失的内容即可。

举个例子：如果你有意让应用内容延伸到挖孔区域下方，可以把 SafeArea 下移，只包裹有必要保护的内容，而让应用其余部分铺满全屏。
补充说明：**AppBar 组件默认就是这么实现的**，因此它可以延伸到系统状态栏下方。
这也是官方推荐用 SafeArea 包裹 Scaffold 的 body，而非包裹整个 Scaffold 的原因。

SafeArea 以通用的方式保证应用内容不会被截断，即便未来市面上出现外形、挖孔样式各异的新设备，你的应用也能从容适配。
